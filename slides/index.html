Consumer-driven contract testing

Problem

Consider Consumer-Driven Contracts as a Design Pattern.

Problem statement:

How can a web service API reflect its clients' needs
 while enabling evolution
 and avoiding breaking clients?

[1] Ian Robinson, Consumer-Driven Contracts, Service Design Patterns

Options

How can we prevent breaking changes to services?

  1. Option 1: service providers create APIs that attempt to address all current and future client needs.
  2. Option 2: service providers write tests based on an interpretation of how clients might use the service.
  3. Option 3: clients provide documentation to providers specifying how they expect to use the service.
  4. Option 4: something else?

[1] Ian Robinson, Consumer-Driven Contracts, Service Design Patterns

Solution

  • Client developers write automated integration tests that express the client's expectations of a service API.
  • These tests are accepted by the service owner, who incorporates them into the service's test suite. (They enter into a contract.)

The set of integration tests received from all existing clients represents the service's obligations to its client base.

[1] Ian Robinson, Consumer-Driven Contracts, Service Design Patterns

Context

Consumer-Driven Contracts don't make sense everywhere.

They should be used when:

  1. A service has several clients, each with different needs.
  2. Service owners know who their clients are, and
  3. Client developers are able to communicate their expectations of the service's API to service owners.
